# 桌面上下文 AI 老师助手 PRD v0

## 1. 产品一句话定义

一个面向 Windows 学习者的常驻桌面 AI 老师助手：用户在电脑屏幕上遇到看不懂、不会、想进一步搜索或需要解释的信息时，可通过快捷键截图快速提问，并在屏幕上的小窗口中获得讲解、搜索结果或后续对话；界面右下角保留一个轻量角色图标，提供存在感和快速入口。

## 2. 先把概念切开

当前想法里包含 3 个元素：

- `AI 老师`：核心价值，负责解释、拆解、启发和辅导。
- `AI 助手`：能力层，负责搜索、提取、整理和少量简单动作。
- `AI 宠物/角色图标`：表现层，负责存在感、入口和轻量情绪连接。

这三者不该是并列定位，而应是层级结构。

本 PRD 的判断：

- **核心产品** 应该是 `桌面上下文 AI 老师`。
- `助手` 不是另一个产品，而是老师调用外部能力的执行层。
- `宠物/角色图标` 不是核心价值，而是常驻桌面的存在感入口。

如果这三层关系不分清，结果大概率会是：

- 老师价值不够硬。
- 助手能力边界失控。
- 角色形象沦为干扰物。

最后什么都像，但什么都不成立。

### 2.1 建议采用的产品结构

- `核心承诺`：帮用户看懂当前屏幕上的内容。
- `主要交互`：截图提问，原地讲解，必要时继续追问。
- `常驻形态`：一个轻量角色图标，表示产品在线且可随时唤起。
- `能力补充`：当模型自身知识不足、需要最新信息或外部工具时，再按需调用 skill。

换句话说：

- 用户买单的不是“桌面上有个宠物”。
- 用户买单的也不是“什么都能做的 agent”。
- 用户买单的是“我看不懂时，它就在旁边，而且真能讲明白”。

## 3. 背景与问题

用户在电脑上工作或学习时，会频繁遇到以下问题：

- 看不懂一段文字、代码、报错、图表、公式或界面提示。
- 想快速确认一个概念、名词或知识点。
- 需要继续搜索相关资料，但不想离开当前工作流。
- 复制文本不方便，或信息存在于图片、视频帧、PDF、软件界面中。

当前替代方案的问题：

- 切去浏览器或聊天工具会打断任务上下文。
- 复制粘贴存在成本，有时甚至无法复制。
- 普通 AI 对话缺乏屏幕上下文，用户还要重新描述。

## 4. 产品目标

### 4.1 核心目标

让用户在 **不离开当前任务上下文** 的前提下，在 **2 步内** 发起问题，并在 **小窗中快速获得可执行、可理解的回答**。

### 4.2 用户价值

- 降低搜索和求助的上下文切换成本。
- 提高“看不懂时”的解决速度。
- 让 AI 不只是回答，还能按老师式结构解释。
- 通过常驻角色图标提供轻量存在感，而不打扰工作流。

### 4.3 商业与产品假设

- 高频问题不是“我要跟 AI 聊天”，而是“我眼前这玩意儿到底是什么，你给我讲明白”。
- 如果产品能围绕屏幕上下文提供即时帮助，它比通用聊天框更接近真实工作流。
- 角色图标能提升召回和使用频次，但前提是核心讲解质量足够高。
- 用户愿意为“更少切换、更快理解、更贴着当前任务”的体验买单或持续使用。

## 5. 目标用户

### 5.1 初始目标用户

MVP 首个目标用户已经明确：

- `学习者/自学者`：在电脑上读资料、看英文、上网课、做题、查概念时，频繁遇到“看不懂但又不想切出去搜”的场景。

这个选择不是拍脑袋，而是来自最初始、最真实的需求来源：**创始人自己就是目标用户**。

这有优势，也有风险：

- 优势：问题真实，能快速迭代。
- 风险：容易把个人习惯误当成普遍需求。

所以后续验证不能只问“我自己觉得顺不顺手”，而要看其他学习者是否也愿意高频使用。

### 5.2 推荐切入人群

建议切入：**会频繁遇到陌生内容、又愿意主动提问的桌面学习者**。

原因：

- 场景高频。
- 价值明确。
- 反馈可量化。
- 比“泛陪伴宠物”更容易验证 PMF。
- 与“右下角常驻角色 + 快捷键截图求助”的交互天然契合。

## 6. 关键使用场景

### 6.1 P0 场景

1. 用户看到一段看不懂的英文/专业术语，截图后让 AI 用中文解释。
2. 用户看到一段报错信息或代码片段，截图后询问原因和排查思路。
3. 用户看到图表、公式、流程图或 UI 页面，截图后请求拆解说明。
4. 用户看到一个名词或现象，要求 AI 帮忙搜索并给出带来源的简要结论。

### 6.2 P1 场景

1. 用户在解释结果基础上继续追问，形成多轮对话。
2. 用户要求 AI 以“老师模式”分步骤讲解，并给出下一步建议。
3. 用户通过语音而非键盘发起提问。

### 6.3 暂不纳入

1. 持续监听整个屏幕并主动打断用户。
2. 自动替用户操作电脑。
3. 以宠物养成、情绪陪伴为主目标的玩法。

## 7. MVP 产品边界

### 7.1 必须做

- 全局快捷键唤起截图/选区。
- 支持截取屏幕区域或当前窗口。
- 支持对截图内容进行 OCR + 视觉理解。
- 支持用户在截图后输入一个简短问题，也支持“直接解释这张图/这段内容”。
- 返回结果以桌面侧边小窗形式展示，不强制切走当前界面。
- 提供常驻但低干扰的角色图标，默认位于屏幕右下角，可作为入口、状态提示和快速召回点。
- 小窗支持继续追问、关闭、收起、固定位置。
- 联网搜索支持两种触发方式：
  - 用户显式点击或表达“查资料/搜索/最新信息”。
  - 模型在白名单内自动判断必须联网时触发。
- 当发生联网或工具调用时，界面必须明确提示。
- 支持历史会话归档，用户可从历史列表中重新打开并继续对话。

### 7.2 可以晚一点做

- 历史记录检索、标签与摘要整理。
- 解释风格切换（简洁模式、老师模式、考试模式等）。
- 语音输入/语音播报。
- 更丰富的角色形象、拟人化动画、宠物状态系统。

### 7.3 不该在 MVP 做

- 主动监控所有屏幕内容并实时分析。
- 复杂的 AI agent 自动执行任务。
- 社交/社区功能。
- 太强的“宠物”叙事，导致工具价值被稀释。

## 8. 用户流程

### 8.1 主流程

1. 用户在电脑上遇到问题。
2. 看到右下角常驻角色图标，确认助手在线可用。
3. 按下全局快捷键。
4. 框选屏幕区域或选择当前窗口。
5. 确认截图并补充问题，或直接点击“解释一下”。
6. 系统分析截图内容。
7. 在屏幕侧边弹出回答小窗。
8. 用户可继续追问、固定窗口、关闭窗口，或发起搜索。

### 8.2 搜索增强流程

1. 用户请求“帮我查一下”“找资料”“最新情况是什么”。
2. 系统识别这是需要联网的任务。
3. 返回简要结论 + 来源列表。
4. 用户可追问“结合截图再解释一次”。

## 9. 核心需求

### 9.1 功能需求

#### FR-1 截图触发

- 系统必须支持全局快捷键触发。
- 用户必须能选择截图区域。
- 系统应支持重复截图的低摩擦流程。

#### FR-2 上下文理解

- 系统必须能识别截图中的文字、界面、图像和基础结构信息。
- 当 OCR 质量不足时，系统应允许用户补充文字问题。
- MVP 输入模式默认是 `截图 + 可选文本问题`。
- 语音输入可以进入后续版本，但不应仅因为底层模型支持音频就强行进入 MVP。

#### FR-3 回答呈现

- 回答必须在小窗内展示。
- 小窗必须支持置顶、拖拽、收起和关闭。
- 回答应优先简洁，再允许展开详细解释。
- 角色图标必须始终低干扰，不得遮挡系统关键 UI 或频繁主动吸引注意力。

#### FR-4 教学式解释

- 系统应支持默认“老师式讲解”回答策略：
  - 先说这是什么。
  - 再说为什么重要。
  - 再说如何理解或下一步怎么做。
- 当用户问技术问题时，应优先给出排查或思考路径，而不只是结论。
- 默认回答风格按“老师式讲解”执行；后续版本再考虑允许用户切换为更简洁的模式。

#### FR-5 能力路由

- 系统默认应优先使用模型自身能力直接回答。
- “模型自己选择是否调用 tools/skills”可以成立，但前提是产品先定义允许调用的范围，而不是无限制放权。
- 联网触发策略采用 `自动路由 + 手动触发并存`：
  - 用户可以显式要求联网搜索。
  - 模型也可以在白名单内自动决定是否联网。
  - 自动联网时必须给出可见状态提示。
- 只有在以下情况才应调用 skill：
  - 用户明确要求联网搜索、查资料、找来源或最新信息。
  - 问题需要外部工具能力，如网页搜索、结构化提取、专用转换或后续可执行动作。
  - 模型对截图内容理解置信度不足，直接回答风险高于补充观察或调用外部能力。
- 当调用 skill 时，系统应让用户感知到“正在搜索/正在调用工具”，避免黑箱感。

#### FR-5.1 Skill 白名单

- `skill 白名单` 的意思不是“模型会不会用 skill”，而是“**第一版产品允许模型调用哪些 skill，不允许调用哪些 skill**”。
- 这是一条产品控制线，用来约束复杂度、时延、权限和错误范围。
- 没有白名单，后果通常是：
  - 模型过度调用工具，导致回答变慢。
  - 工具边界不断膨胀，产品从老师助手滑向泛 agent。
  - 出错时难以定位，到底是模型错、工具错，还是路由错。

MVP 建议白名单：

- `联网搜索 skill`：当用户要求最新信息、资料、来源时使用。
- `网页打开与来源提取 skill`：当需要基于搜索结果或用户提供链接做总结时使用。

MVP 不建议放开的 skill：

- 系统控制类 skill，如自动点按钮、自动操作电脑。
- 高权限文件系统类 skill，如任意扫描本地文件。
- 多步 agent 类 skill，如自行规划并执行复杂任务链。

结论：

- 模型可以在白名单内自主选择。
- 模型不能越过白名单自作主张。
- 白名单越小，第一版越容易做稳。

#### FR-6 搜索增强

- 系统必须区分“解释当前截图”和“联网查资料”。
- 需要联网时，应展示来源，避免无来源硬编。

#### FR-7 多轮追问

- 用户应可基于当前截图上下文继续提问。
- 上下文会话应至少在当前小窗生命周期内保留。

#### FR-8 会话归档与上下文生命周期

- 关闭当前聊天窗口后，运行时上下文应被清空，不保留为后台常驻记忆。
- 系统应保留历史会话存档，用户可手动重新打开某条历史对话并继续交流。
- 产品不应在不同会话之间自动建立“长期用户记忆”或隐式画像。
- 用户必须可以删除单条历史会话。

#### FR-9 隐私控制

- 每次上传截图前，用户必须有明确触发动作。
- 产品必须明确告知截图可能包含敏感信息。
- 用户应可一键删除本次会话内容。

### 9.2 非功能需求

- 首次回答延迟目标：
  - 纯解释场景 < 5 秒。
  - 联网搜索场景 < 8 秒。
- 小窗不应明显遮挡用户当前核心工作区域。
- 全局快捷键和截图流程必须稳定，不可频繁失焦或卡死。
- 崩溃恢复和错误提示必须明确，不能静默失败。

## 10. 成功指标

### 10.1 早期验证指标

- 用户从触发截图到看到首条回答的中位耗时。
- 每用户每周截图提问次数。
- 截图提问后继续追问的比例。
- 回答后被立即关闭的比例。
- 用户主观评分：这次回答是否真正帮你理解了问题。
- 用户主观评分：角色图标是否有存在感但不烦。
- 学习者在一次学习流程内连续使用 2 次以上的比例。

### 10.2 判断产品是否有价值的信号

- 用户不是把它当普通聊天框，而是明确在“遇到屏幕问题时”反射性打开。
- 截图入口的使用频次高于纯文本输入。
- 用户会在同一工作流里连续多次使用，而不是尝鲜后流失。

## 11. 风险与压力测试

### 11.1 最大产品风险

- **定位风险**：如果“助手、老师、宠物”三种定位不拆开，产品会失焦。
- **性能风险**：如果截图后等待过久，用户会直接回到浏览器搜索。
- **准确性风险**：截图内容理解不准会迅速破坏信任。
- **隐私风险**：屏幕截图天然可能带来账号、聊天记录、公司数据泄露问题。
- **交互风险**：小窗如果挡事或过于吵闹，会被用户视为干扰物而不是助手。
- **路由风险**：如果过度调用 skill，延迟和复杂度会飙升；如果死撑模型直答，又会在需要来源和最新信息时胡说。

### 11.2 最大战略风险

- 这不是一个新奇点子，市场里早就存在“桌面 AI + 截图 + 解释”的方向。
- 如果没有更强的场景聚焦、回答质量、速度或交互体验，产品只会沦为另一个入口壳子。

### 11.3 对策

- 先做“桌面上下文解释”这一件事，别做泛 AI。
- 把“老师”作为核心价值，把“助手”作为能力补充，把“角色图标”作为存在感入口。
- 强控时延和 UI 打扰度。
- 为 skill 设清晰调用边界，默认模型直答，必要时再用工具。
- 对联网内容默认附来源，对无法判断的内容明确说不确定。

## 12. 版本规划

### V0

- Windows 平台
- 右下角常驻角色图标
- 全局快捷键截图
- 截图 + 文本提问
- 小窗回答
- OCR + 视觉理解
- 简单追问
- 联网搜索与来源提取白名单
- 历史会话归档与续聊

### V1

- 回答风格切换
- 历史记录搜索与整理
- 语音输入/输出

### V2

- 更拟人的角色和动画表现
- 个性化学习档案
- 跨设备同步或更长期的学习记忆

## 13. 当前建议的产品结论

MVP 不要做“宠物玩法”。

MVP 应定义为：

> 一个带有轻量角色存在感、以教学式讲解为核心、并能按需调用外部 skill 的桌面 AI 老师助手。

其中：

- `老师` 是核心定位。
- `助手` 是能力层。
- `角色图标` 是表现层。
- `宠物玩法` 是未来可选包装层。

## 14. 已明确决策

1. 首发平台只做 `Windows`。
2. 首个目标用户是 `学习者/自学者`，因为核心需求来自高频真实学习场景。
3. 角色图标默认位于 `屏幕右下角`，提供存在感和快速入口。
4. 核心触发路径是 `快捷键截图 -> 用户确认 -> 交给 AI 解答`。
5. 能力路由采用 `model-first, skill-on-demand`：
   - 默认模型直接回答。
   - 需要最新信息、来源或外部能力时，再由模型在白名单内选择合适的 tool/skill。
6. 回答小窗默认显示在 `屏幕侧边`，不贴着截图区域漂浮。
7. 联网触发采用 `自动判断 + 用户手动触发并存`。
8. 默认回答风格采用 `老师式讲解`。
9. 第一版只开放 `联网搜索 + 网页来源提取` 两类 skill。
10. 当前聊天窗口关闭后清空运行时上下文，但保留 `历史会话归档`，用户可手动续聊。
11. 语音输入严格放到 `V1`，不进入 `V0`。
12. 历史会话归档在 `V0` 默认仅本地存储，不做云同步。
13. 技术方案暂定兼容 `OpenAI` 与 `Qwen` 两类 provider，通过统一模型适配层接入。

## 15. 仍需拍板的问题

1. `OpenAI` 与 `Qwen` 各自首选接入的具体模型 ID 是什么。
2. provider 配置是做统一 OpenAI-compatible 接口，还是分别维护适配器。
3. Windows 桌面壳首版采用什么技术栈。
